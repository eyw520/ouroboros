You are driving a target repo to full adoption of the standards in this repo.
The scripts do the mechanical parts; the judgment calls below are yours.
Work phase by phase; do not skip the classification gate.

## Phase 0 — Classify: should this repo be standardized at all?

Read the target's CLAUDE.md/README **before touching anything**. Decline (and
tell the user why) when the repo is:

- **Machine-written** — commits are generated by automation (memory snapshots,
  session recorders). The convention would fight the auto-committer.
- **A fork of someone else's project** — it follows upstream community
  conventions; stamping ours creates merge conflicts.
- **Explicitly opted out** — a repo whose own docs contradict the standard
  (e.g. "agents never commit — stage nothing and stop at the diff") keeps its
  policy. Its CLAUDE.md wins over AGENTS.md. Apply at most inert files
  (.editorconfig), leave them uncommitted, and report.

## Phase 1 — Survey

1. `<standards>/doctor.sh <target>` — conformance report + a type(scope)
   histogram of the last 100 subjects.
2. Read the target's CLAUDE.md, commit hook, Makefile, and CI workflow.
3. Note every drift: hook vs CLAUDE.md prose vs actual history. **The hook is
   the source of truth; observed history is the tiebreaker when creating or
   changing a hook.** A greenfield repo (no history) follows its stated intent.

## Phase 2 — Choose the config

- Types: what history actually uses (base `feat|fix|chore|clean|revert`; keep
  per-repo extras like `docs`, `refactor`, `exec` that appear in history).
- Scopes: from the histogram. No scopes in history → scopeless (`-s ""`).
  Monorepos may derive scopes dynamically from subdirectories (see the
  commented block in the template hook).

## Phase 3 — Stamp

`<standards>/init.sh [-t types] [-s scopes] [-l python|node|none] [-c] <target>`

- It installs only missing files and wires `core.hooksPath`. `DIFFERS` lines
  are yours to reconcile by hand — never blind-overwrite a repo's own file.
- An existing hook that already matches history stays; replace a hook only
  when it is wrong (e.g. rejects the format the repo's own docs mandate).

## Phase 4 — Integrate docs and Makefile

- Add `@AGENTS.md` on its own line near the top of the target's CLAUDE.md.
- **Dedupe**: delete CLAUDE.md lines now covered by AGENTS.md (generic first
  principles, generic commit discipline). **Keep** every repo-specific nuance
  (extra types/scopes, "a schema snapshot rides the commit", domain
  principles). A one-line pointer ("The rest of the commit discipline lives in
  `AGENTS.md`.") replaces the removed block.
- Makefile verb contract — `check` (the full gate), `fmt`, `hooks`, `dev` —
  mapped onto the repo's *real* toolchain. Alias, don't duplicate (e.g.
  `fmt: lint` when lint already fixes; `dev: setup` when setup exists). If
  the repo's gate is a script, wrap it (`check: ./tests/run.sh`).
- CI: `.github/workflows/gate.yml` = toolchain setup + `make check`, nothing
  else — only when the repo has a GitHub remote.

## Phase 5 — Verify the gate (where CI bugs actually hide)

Run the full gate locally and **check the real exit code** — piping to
`tail` swallows it; use `cmd > log 2>&1; echo $?`.

Fresh-checkout pitfalls that pass locally and fail in CI — probe each:

- **Stale virtualenvs**: after a project rename, venv shebangs point at the
  old path and tools half-work. Rebuild, and expect a newer tool version to
  surface pre-existing errors.
- **Version-sensitive linters**: CI's apt packages are older/noisier than
  local brew (shellcheck!). Pin the CI tool to the version the baseline is
  clean against.
- **Shell globs over gitignored dirs**: `experiments/*/src/` dies under zsh
  on a checkout that lacks them. Use make `$(wildcard)`.
- **Tests referencing local-only paths**: skip-with-reason when absent,
  assert fully when present.
- **Stale lockfiles**: `poetry install`/`npm ci` fail on a lock that drifted
  from its manifest; refresh as its own commit.

A pre-existing red gate: fix it as its **own** commit(s) *before* the adoption
commit, so every commit lands gate-green. Large pre-existing debt (dozens of
type errors): don't absorb it — scope CI down (`make check` without
`typecheck`), leave a comment naming the debt, and report it.

## Phase 6 — Commit

- Follow the **target's** convention (its hook decides). One logical change
  per commit: gate repairs first, then one adoption commit
  (`chore: Adopt the shared engineering conventions.` — scoped form where the
  repo uses scopes).
- Stage your files explicitly; never `git add -A` in a repo with unrelated
  working-tree changes.
- Commit locally and stop. Push only when the operator says push. When you do
  push and CI is new, watch the first run and fix-forward until green.

## Phase 7 — Report

Outcome per repo, drift repaired, debts left (and where they're noted),
skipped repos with reasons, and security flags. **Tracked credential-shaped
files (.pem/.key/secrets) are a loud flag to the operator — rotation plus
history rewrite is their call, never yours.**
